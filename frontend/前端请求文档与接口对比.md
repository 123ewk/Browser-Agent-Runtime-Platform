## 三、问题详细分析

### 🔴 P0 — 后端缺失接口（前端调用会 404）

#### 1. `POST /tasks/:id/messages` — 向 Agent 发送指令

**前端调用**：`lib/api/timeline.ts` → `postTaskMessage(taskId, content, runMode)`

```typescript
// 前端发送
POST /tasks/{taskId}/messages
Body: { content: string, runMode: "yolo" | "semi" }
// 期望响应
ChatMessage { id, taskId, role, content, createdAt }
```

**后端现状**：`app/api/tasks.py` 中没有 `/tasks/{task_id}/messages` 端点。

**影响**：Agent Workspace 中用户无法向正在运行的任务发送指令（如"换一种方式"、"暂停"等），半自动模式（semi）无法工作。

**建议**：后端新增 `POST /tasks/{task_id}/messages` 端点，接收 `{ content, runMode }`，通过 `BrowserTaskRunner.send_command()` 将用户指令转发给 Worker。

---

#### 2. `GET /users/me` — 获取当前用户信息

**前端调用**：`lib/store/auth.ts` 注释中明确提到需要此接口。

```typescript
// 前端期望
GET /users/me
Headers: Authorization: Bearer <token>
// 期望响应
UserOut { id: uuid, username: string, created_at: datetime }
```

**后端现状**：没有 `/users/me` 端点。

**影响**：前端 `deriveUserFromToken()` 只能用 `id: "pending"` 占位，用户信息不完整。如果后续需要用户 ID 做数据关联（如偏好、任务归属），`"pending"` 会导致问题。

**建议**：后端新增 `GET /users/me` 端点，从 JWT 中解析 user_id，返回 `UserOut`。

---

### 🟡 P1 — 响应字段不匹配

#### 3. `GET /tasks/:id` — 任务详情响应字段缺失

**前端期望** (`TaskDetail`)：

```typescript
{
  id, goal, agentName, status, createdAt, updatedAt, costUsd,  // 继承自 TaskSummary
  totalDurationSec, totalTokens, steps, screenshots, skillCalls  // TaskDetail 独有
}
```

**后端实际返回**：

```python
{
  "task_id": task_id,    # ← 字段名不一致！前端用 id，后端用 task_id
  "state": state.value,  # ← 字段名不一致！前端用 status，后端用 state
  "reason": reason       # ← 仅 3 个字段，缺少 7 个字段
}
```

**缺失字段**：
| 前端字段 | 后端是否返回 | 说明 |
|---|---|---|
| `goal` | ❌ | 任务目标 |
| `agentName` | ❌ | Agent 名称 |
| `createdAt` | ❌ | 创建时间 |
| `updatedAt` | ❌ | 更新时间 |
| `costUsd` | ❌ | 花费 |
| `totalDurationSec` | ❌ | 总耗时 |
| `totalTokens` | ❌ | 总 token 数 |
| `steps` | ❌ | 步骤列表 |
| `screenshots` | ❌ | 截图列表 |
| `skillCalls` | ❌ | Skill 调用列表 |

**字段名不一致**：
| 前端 | 后端 | 建议 |
|---|---|---|
| `id` | `task_id` | 后端统一用 `id`，与列表接口保持一致 |
| `status` | `state` | 后端统一用 `status`，与列表接口保持一致 |

**建议**：后端 `GET /tasks/{task_id}` 应从数据库查询完整任务信息，返回与 `GET /tasks` 列表项一致的字段结构，并补充 `steps`、`screenshots` 等详情字段。

### 🟡 P2 — 前端未对接的后端接口

#### 4. 偏好模块 (`/preferences/*`) — 前端完全未对接

后端已实现 4 个偏好接口：
- `GET /preferences` — 全量查询
- `POST /preferences` — 创建/更新
- `DELETE /preferences/{id}` — 删除

**影响**：Settings 页面（`app/(workspace)/settings/page.tsx`）无法管理用户偏好，Agent 执行时也无法加载用户长期记忆注入 system prompt。

**建议**：前端新增 `lib/api/preferences.ts` + `types/preferences.ts`，对接后端 4 个接口。

**建议**：如果 TopBar 需要显示后端健康状态，新增 `lib/api/health.ts`。

### 🔵 P3 — 类型/命名不一致（不影响运行但增加维护成本）

#### 6. `TaskStatus` 枚举值差异

**前端定义**（`types/task.ts`）：
```
pending | running | waiting_confirm | paused | stopping | completed | failed | cancelled
```

**后端定义**（`runtime/protocol/types.py` TaskState）：
```
PENDING | RUNNING | STOPPING | COMPLETED | FAILED | CANCELLED
```

**差异**：
| 前端 | 后端 | 状态 |
|---|---|---|
| `pending` | `PENDING` | ✅ 对应（大小写） |
| `running` | `RUNNING` | ✅ 对应 |
| `waiting_confirm` | — | ⚠️ 后端无此状态 |
| `paused` | — | ⚠️ 后端无此状态 |
| `stopping` | `STOPPING` | ✅ 对应 |
| `completed` | `COMPLETED` | ✅ 对应 |
| `failed` | `FAILED` | ✅ 对应 |
| `cancelled` | `CANCELLED` | ✅ 对应 |

后端返回的 status 是小写字符串（`state.value`），前端用小写匹配，大部分 OK。但 `waiting_confirm` 和 `paused` 前端预留了但后端未实现，前端渲染这两种状态时会是空逻辑。

---

#### 7. `TimelineStep.kind` 枚举差异

**前端定义**（`types/timeline.ts`）：
```
think | tool | observe | human | complete
```

**后端 `_step_kind_from_action()` 返回**：
```
tool | observe
```

**差异**：后端只返回 `"tool"` 或 `"observe"`，前端定义了 5 种 kind。`"think"`、`"human"`、`"complete"` 后端不会返回，前端渲染这三种 kind 的 UI 分支永远不会触发。
